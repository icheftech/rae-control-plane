# PostgreSQL Test Infrastructure Design Spec

**Date:** 2026-06-13
**Status:** Approved by Leroy Brown
**Scope:** Replace SQLite-based test fixtures with real PostgreSQL via Testcontainers; produce `TEST_INFRA_REVIEW.md`

---

## 1. Problem Statement

The backend test suite (`backend/tests/`) uses an in-memory SQLite database. The application models use PostgreSQL-specific column types (`UUID`, `JSONB`, `ARRAY`). The prior workaround — `_patch_metadata_for_sqlite()` in `conftest.py` — mutates `Base.metadata` in-place at runtime to swap those types for SQLite equivalents. This has two failure modes:

1. **Type mutation failures:** Any new PostgreSQL-specific type added to a model silently breaks tests unless the patch function is also updated.
2. **Semantic drift:** SQLite's constraint enforcement, JSONB operator support, and UUID behavior differ from PostgreSQL, so tests can pass on SQLite while the production path fails.

---

## 2. Approach: Testcontainers + PostgreSQL 15-Alpine

**Chosen:** Option A — `testcontainers[postgresql]`

Start a real PostgreSQL 15-Alpine container (matching `docker-compose.yml: postgres:15-alpine`) scoped to the pytest session. Tests run against actual PostgreSQL semantics. No SQLite type patches. No local PostgreSQL installation required.

**Rejected alternatives:**
- `pytest-postgresql` — requires a local `pg_ctl` binary; not installed on this machine.
- SQLite + improved patch — never truly tests PostgreSQL behavior.

## 4. Fixture Architecture (`backend/tests/conftest.py`)

Replace the entire SQLite approach with three fixture layers:

### Layer 1 — `postgres_container` (session scope)
```python
@pytest.fixture(scope="session")
def postgres_container():
    with PostgresContainer("postgres:15-alpine") as pg:
        yield pg
```
One container per pytest session. Startup ~2–3 s on first run after image pull; ~1 s on cached runs.

### Layer 2 — `test_engine` (session scope)
```python
@pytest.fixture(scope="session")
def test_engine(postgres_container):
    engine = create_engine(postgres_container.get_connection_url())
    import app.db.models  # ensure all models register on Base.metadata
    Base.metadata.create_all(bind=engine)   # ← temporary shortcut (see §5)
    yield engine
    Base.metadata.drop_all(bind=engine)
    engine.dispose()
```

### Layer 3 — `test_db` (function scope, nested transaction)
```python
@pytest.fixture(scope="function")
def test_db(test_engine):
    connection = test_engine.connect()
    transaction = connection.begin()
    session = Session(bind=connection)
    nested = connection.begin_nested()   # SAVEPOINT

    @event.listens_for(session, "after_transaction_end")
    def restart_savepoint(session, transaction):
        if transaction.nested and not transaction._parent.nested:
            session.expire_all()
            nested = connection.begin_nested()

    yield session

    session.close()
    transaction.rollback()   # outer rollback undoes everything
    connection.close()
```

**Why nested transactions (SAVEPOINT)?** Application code — and endpoint tests via `TestClient` — may call `session.commit()`. A bare `connection.begin()` + rollback would still see those commits. The `begin_nested()` (SAVEPOINT) pattern re-issues a new SAVEPOINT after each nested commit, so the outer transaction sees all writes but the final rollback erases them. No data bleeds between tests.

### `client` fixture
The existing `client` fixture wires `test_db` into FastAPI's `get_db` override; its body is unchanged. The `_patch_metadata_for_sqlite` function and `StaticPool` import are deleted.

## 5. Schema Setup: `Base.metadata.create_all()` as Temporary Shortcut

The single Alembic migration (`migrations/versions/001_initial_schema.py`) has an empty `upgrade()` body — it's a placeholder stub, not an autogenerated migration. Running Alembic against the test container would create an empty schema.

**Decision:** Use `Base.metadata.create_all()` for test setup now. Document as temporary.

**Recommended follow-up (out of scope for this task):** Run `alembic revision --autogenerate -m "Initial schema"` against a live PostgreSQL instance to generate a real migration, then switch the test fixture to run `alembic upgrade head` against the container URL. At that point, `create_all` can be removed and migrations become the single source of truth for schema in both tests and production.

## 6. `pytest.ini` — No Changes

The existing `--cov-fail-under=80` gate and all markers stay. Two test runs are documented separately:

- **Run 1 (functional):** `pytest --no-cov -q` — confirms tests pass/fail independently of coverage.
- **Run 2 (coverage):** `pytest` with the full `pytest.ini` `addopts` — determines coverage gate result.

## 7. Production Code Changes

**None.** Models, migrations, `database.py`, `base.py`, API routes, and `app/main.py` are untouched. The change is entirely within `backend/tests/conftest.py` and `backend/requirements.txt`.
